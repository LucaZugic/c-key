# Commit Conventions

Commits are the permanent record of how the codebase evolved. Good commits make debugging, reviewing, and reverting easier. Bad commits make everything harder.

## Core Principles

### Small and Focused

Each commit does one thing. If you find yourself writing "and" in the commit message, split it.

### Independently Valid

Each commit should:
- Build successfully
- Pass all tests
- Not break the app (even temporarily)

This enables `git bisect` and makes reverting safe.

### Tell Why, Not What

The diff shows what changed. The message explains why.

## Format

```
<type>: <subject>

<body>
```

### Types

| Type | Use For |
|------|---------|
| `feat` | New feature or capability |
| `fix` | Bug fix |
| `refactor` | Code restructuring without behavior change |
| `test` | Adding or modifying tests |
| `docs` | Documentation changes |
| `chore` | Build, tooling, or maintenance |

### Subject Line

- Imperative mood: "Add filter" not "Added filter" or "Adds filter"
- Lowercase after type prefix
- No period at end
- Maximum 72 characters
- Specific: "add DistanceBetween filter" not "add new filter"

### Body (Optional)

- Separated from subject by blank line
- Explains why the change was made
- Can include context, trade-offs, or links to issues
- Wrap at 72 characters

## Examples

### Good Commits

```
feat: add DistanceBetween filter for rule evaluation
```
Simple feature, self-explanatory.

```
fix: handle nil gear in activity mapping

Strava API returns null gear_id when no gear is assigned.
Previously we crashed on force-unwrap. Now we map to nil
and let rules handle the missing-gear case.
```
Bug fix with context explaining the issue.

```
refactor: extract token refresh into dedicated method

Preparation for adding automatic retry on 401.
No behavior change.
```
Refactoring with explanation of intent.

```
test: add boundary cases for DistanceBetween filter

Covers min boundary, max boundary, below range, above range.
Parameterized to reduce duplication.
```
Test addition with coverage summary.

### Bad Commits

```
Update code
```
What code? Why? Useless.

```
Fix bug
```
Which bug? This helps no one.

```
WIP
```
Never commit work-in-progress. Finish it or stash it.

```
Add new feature and fix some bugs and update docs
```
Three things = three commits.

```
feat: Add DistanceBetween filter.
```
Capitalized subject, period at end. Minor but sloppy.

```
feat: add filter

Co-authored-by: Claude <noreply@anthropic.com>
```
**Never include AI attribution.** The author is the developer. This is non-negotiable.

---

## AI Attribution: Forbidden

**Do not add any of the following to commit messages:**

- `Co-authored-by: Claude`
- `Generated by Claude`
- `AI-assisted`
- Any reference to Anthropic, Claude, or AI tooling

The developer is the author. Tools are tools.

## Commit Workflow

1. Stage related changes: `git add -p` for precision
2. Review what's staged: `git diff --staged`
3. Write message following format
4. Commit
5. Verify: `git log -1` to review

### Amending

Amend only if:
- You just committed and noticed a typo
- Pre-commit hooks modified files

Never amend commits that are pushed or authored by others.

### Squashing

Interactive rebase to squash is fine for cleaning up local history before push. Don't squash after pushing.

---

## Commit Frequency

Commit often. Small commits are better than big ones.

Typical flow for a feature:
1. `test: add failing test for X`
2. `feat: implement X`
3. `refactor: extract Y from X`
4. `test: add edge cases for X`

Four commits for one feature is fine. One giant commit is not.

---

## Integration with TDD

The TDD loop naturally produces good commits:

| TDD Step | Commit Type |
|----------|-------------|
| Write failing test | `test:` |
| Make test pass | `feat:` or `fix:` |
| Refactor | `refactor:` |

You might combine "write test" and "make it pass" into one `feat:` commit if they're small. But keep refactoring separate — it makes the history clearer.

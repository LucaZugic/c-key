# Commit Guidelines

Commits in c-key follow Conventional Commits format. Each commit is small, focused, and independently valid.

## Format

```
<type>(<scope>): <subject>

<body>

<footer>
```

### Type (Required)

- `feat`: New feature
- `fix`: Bug fix
- `docs`: Documentation only
- `style`: Formatting, no code change
- `refactor`: Code change that neither fixes a bug nor adds a feature
- `test`: Adding or updating tests
- `chore`: Tooling, build, dependencies

### Scope (Optional)

The module or area affected: `domain`, `strava`, `shortcut`, `ci`, etc.

### Subject (Required)

- Imperative mood: "add" not "added" or "adds"
- Lowercase first letter
- No period at the end
- Maximum 72 characters

### Body (Optional)

Explain *why* the change was made, not *what* changed (the diff shows what). Wrap at 72 characters.

### Footer (Optional)

Reference issues: `Closes #123` or `Fixes #456`.

## Examples

### Good Commits

```
feat(domain): add DistanceLessThan filter evaluation

The filter compares activity distance against a threshold and returns
true if the distance is strictly less than the threshold.
```

```
fix(strava): handle 429 rate limit with exponential backoff

Strava returns 429 when rate limited. Previously we failed immediately.
Now we wait and retry up to 3 times with exponential backoff.

Closes #42
```

```
refactor(domain): extract filter evaluation into separate module

Moved filter logic from rules.ts to filters.ts for better organization.
No behavior change.
```

```
test(domain): add edge cases for DistanceBetween filter

Cover boundary conditions where distance equals min or max threshold.
```

```
chore: update typescript to 5.4
```

### Bad Commits

```
// Too vague
fix: bug fix

// Past tense
Added new feature

// Too many changes in one commit
feat: add all filters and actions and rule evaluation

// AI attribution (NEVER DO THIS)
feat: add feature

Co-authored-by: Claude <noreply@anthropic.com>
```

## No AI Attribution

**Never include AI attribution in commits.** No `Co-authored-by: Claude`. No "generated by AI". No references to Anthropic, Claude, or any AI assistant. The author is the developer. This is non-negotiable.

## Commit Atomicity

Each commit should:

1. **Build**: `npm run typecheck` passes
2. **Lint**: `npm run lint` passes
3. **Test**: `npm test` passes
4. **Be focused**: One logical change

If a change requires multiple logical steps, make multiple commits. "Add filter type" and "add tests for filter type" can be one commit (TDD produces them together). "Add filter type" and "update documentation" should be two commits.

## Commit Frequency

Commit often. A good pace is one commit per TDD cycle (one new test + implementation + refactor). Avoid large commits that accumulate hours of work.

If you realize a commit is too large, use `git add -p` to stage partial changes and split into multiple commits.

## Commit Messages for Reviewers

Write commit messages for a future reader who needs to understand why the codebase looks the way it does. Six months from now, "fix bug" is useless. "Fix off-by-one in distance comparison causing 5km runs to match <5km filter" tells a story.

## Amending and Rebasing

Before pushing:
- Amend the last commit to fix typos or small mistakes: `git commit --amend`
- Rebase to clean up work-in-progress commits: `git rebase -i`

After pushing:
- Do not rewrite history on shared branches
- Create new commits to fix issues

On `main`, prefer a clean history. Squash-merge feature branches if they have messy intermediate commits.
